    pipguard 现状评估与改进建议（功能与检查能力）¶

    评估时间：2026-04-13 评估范围：当前仓库主流程（download → extract → AST scan → gated install）

    1) 当前做得好的地方（功能层）¶

    1. 默认阻断安装期高危行为：
    2. 对 install-hook 范围（setup.py/pyproject.toml/.pth）的网络调用、动态执行、shell 执行等行为设置为 HIGH/CRITICAL，默认阻断。
    3. 对 .pth autorun 攻击有针对性防护：
    4. 单独扫描 .pth，检测可执行语句，直接 CRITICAL。
    5. 具备 TOCTOU 防护思路：
    6. 安装阶段使用 pip install --no-index --find-links 从“已扫描目录”安装，避免二次拉取被替换。
    7. 对二进制盲区有显式提示：
    8. 二进制扩展（.so/.pyd/.dylib）会被标记为 LOW；纯二进制轮子会触发 MEDIUM。
    9. 工程质量基础较稳：
    10. 当前测试通过率高（187 passed, 1 skipped），说明已有行为在现有规则下较稳定。

    2) 当前不足（检查能力层）¶

    2.1 可被绕过的“代码语义”攻击¶

    1. 别名/包装调用绕过（高优先级）

    2. 当前网络检测依赖固定函数名（如 requests.get / urllib.request.urlopen）， 若攻击者写成：

      • import requests as r; r.get(...)
      • fn = requests.get; fn(...)
      • getattr(requests, "get")(... ) 很可能漏报。

    3. 反射与动态属性访问绕过

    4. 如 getattr(os, "system")("...")、__import__("socket") 后再动态调用。

    5. 目前主要覆盖直接调用路径，反射链可规避规则。

    6. 敏感路径检测偏字面量匹配

    7. 目前对凭据路径更多基于字符串常量包含关系， 对 os.path.join(os.path.expanduser("~"), ".ssh", key_name) 这类组合构造不够强。

    8. 命令执行检测范围有限

    9. 已覆盖 os.system/os.popen、subprocess.*(shell=True)，

    10. 但对 subprocess.Popen(["bash","-c", ...])（shell=False）或 pty/ctypes 触发执行，覆盖不足。

    11. 大文件跳过策略可被利用

    12. 单文件 >1MB 直接跳过 AST 扫描，攻击者可将恶意逻辑藏入超大脚本文件实现规避。

    2.2 包管理与供应链元数据攻击面¶

    1. requirements 生态支持面偏窄

    2. 当前对 VCS/直接 URL/本地路径依赖基本是“不支持即退出”， 在真实企业依赖结构中常见，导致工具“用不起来”，进而被绕过（团队直接不用 pipguard）。

    3. 缺少完整性校验闭环

    4. 当前核心是“行为扫描”，但缺失：
      • 强制 hash-pinning（--require-hashes 级别）
      • 与锁文件/制品仓库的一致性验证

    5. 这会让“同名不同内容”的供应链攻击仍有操作空间。

    6. 对 wheel 内元数据和构建后门检查不足

    7. 如 entry_points, console_scripts, .dist-info 元数据滥用、 或安装后触发链（首次 import 执行）缺乏专项规则。

    2.3 二进制与原生扩展盲区¶

    1. 二进制代码不可见（项目已识别但未真正解决）
    2. 目前只是“提示风险”，无法分析 C/Rust 扩展真实行为。
    3. 纯二进制包仅 MEDIUM
    4. 对高安全场景（CI/CD、生产镜像构建）不够严格，建议可配置为默认阻断。

    2.4 运行时攻击能力未覆盖¶

    1. 仅静态扫描，缺少行为沙箱
    2. 无法捕捉“静态看起来安全，但运行期解密/下载二阶段 payload”的攻击。
    3. 缺少系统调用/网络/DNS 实时拦截
    4. 不能在安装进程中基于能力策略阻断如 connect/open/read ~/.ssh 等实际操作。

    3) 典型“仍可能防不住”的 attack 列表¶

    按现实风险优先级排序：

    1. Alias + 反射混淆 exfiltration

    2. 使用别名/getattr/变量间接调用网络与文件读取，绕过函数名匹配。

    3. 大型单文件载荷（>1MB）隐藏恶意逻辑

    4. 利用扫描上限跳过解析。

    5. 纯二进制 wheel 内置恶意逻辑

    6. 通过 native 扩展直接执行系统调用/网络发送。

    7. 安装后延迟触发（post-install / first-import）

    8. 不在 install hook 里做坏事，而在首次 import 时执行。

    9. 利用合法能力组合达成恶意目标

    10. 比如不直接访问敏感路径，而先调用系统工具间接读取并 exfiltrate。

    11. 多层字符串拼接 + 解码 + exec 链

    12. 超出当前 b64 + exec 的单层模式规则。

    4) 改进建议（按落地阶段）¶

    阶段 A（1~2 周，可快速提升检出率）¶

    1. 引入“简单数据流/别名追踪”
    2. 在 AST 层维护轻量符号表：
      • 跟踪 import x as y
      • 跟踪 fn = requests.get 这类一跳赋值

    3. 将调用点还原到 canonical API，再匹配规则。

    4. 扩展危险调用规则库

    5. 增补 getattr(..., "system")、Popen(["bash","-c"])、shlex/pty/ctypes 常见链路。

    6. 优化大文件策略

    7. 不再“直接跳过”，改成：

      • 分块抽样 + 关键 token 扫描
      • 并给出 MEDIUM/HIGH “未完全扫描”告警。

    8. 提高 install-hook 严格性（默认 deny）

    9. install-hook 里除白名单操作外，网络/命令执行默认 CRITICAL。

    阶段 B（2~6 周，构建企业可用能力）¶

    1. 增强供应链完整性功能
    2. 支持 requirements hash 校验模式；

    3. 生成并验证 lock 清单（包名+版本+hash）。

    4. 引入可配置策略文件（policy-as-code）

    5. 例如 pipguard.toml：

      • CI 中 binary_only = block
      • 允许域名白名单
      • 禁止 install-hook 网络。

    6. 支持更多依赖来源但保持安全默认

    7. VCS/URL 依赖可“受限支持”：必须 pin commit/hash，否则阻断。

    阶段 C（中期方向，能力跃迁）¶

    1. 运行时沙箱（seccomp/eBPF/容器隔离）

    2. 将安装过程放入隔离环境，按能力策略拦截：

      • 文件读（~/.ssh, ~/.aws）
      • 网络连接（任意外连或非白名单域名）
      • 子进程执行。

    3. 二进制扩展强化审计

    4. 对 .so/.pyd 做静态特征扫描（导入符号、可疑字符串、网络 API 引用）；
    5. 与签名可信源（内部仓库）结合。

    5) 建议的目标指标（便于衡量改进效果）¶

    1. 检测能力指标
    2. 恶意样本集检出率（TPR）

    3. 绕过样本集检出率（Alias/Reflection/Obfuscation）

    4. 误报指标

    5. Top 100 常用包误报率（FPR）

    6. 允许列表依赖比例（越低越好）

    7. 工程指标

    8. 扫描耗时（P50/P95）
    9. CI 接入成功率（项目可落地率）

    6) 总结（给当前项目的判断）¶

    • 定位判断：pipguard 目前是一个“方向正确、基础扎实”的 Phase-1 静态防线， 在防范“明显安装期恶意行为”方面有实际价值。
    • 核心短板：对“语义绕过、二进制 payload、运行时动态行为”防护不足。
    • 优先级建议：先做 别名/数据流增强 + 大文件策略 + policy 化， 再推进运行时沙箱，能在不大幅牺牲性能的情况下显著提升真实防护强度。
